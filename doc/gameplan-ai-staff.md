# Cookbookverse — AI Staff Gameplan

> The kitchen runs itself. You own it.
>
> This document maps the AI Staff concept to concrete build phases. Each phase ships real value and stands on its own.

---

## Where We Start

A lot of the raw capability already exists:
- Court of Chefs review pipeline → **this becomes Theo**
- Admin recipe generator → **this becomes Marco**
- Dietary adaptation in AI chat → **this is Nadia's existing work**

The goal is not to rebuild — it's to formalize, connect, and automate what's already running.

---

## Phase 1 — Give the Crew Names and Memory

*Goal: The system knows who its staff are. Every AI action is attributed to a persona. The admin dashboard starts showing crew activity instead of anonymous API calls.*

### What Ships
- **Staff config** — a static config object (or DB table) defining each crew member: name, role, model, prompt template ID, and token budget share
- **`staff_activity` table** — every AI call logs: who ran it, what it did, input/output token count, outcome (PASS / FLAG / REJECT / CREATED / REVIEWED), timestamp, and target recipe ID
- **Rebrand the admin dashboard** — submission reviews show "Reviewed by Theo" with his structured report. Recipe generation shows "Generated by Marco." These are display changes — the underlying APIs don't change yet
- **Staff Activity Log panel** in the admin dashboard — a simple table: who ran, what, when, outcome

### What This Unlocks
- You see exactly what's happening and who did it
- Token usage per persona starts accumulating — Ellis needs this data in Phase 2
- Foundation for everything else: all future phases write to `staff_activity`

### DB Change
```sql
staff_activity (
  id, persona, action_type, recipe_id, tokens_input, tokens_output, 
  outcome, notes, created_at
)
```

## Phase 2 — Ellis: The Eyes of the Kitchen

*Goal: Ellis can see the state of the platform and surface a weekly digest. You start getting a coherent picture of what's happening without having to look for it.*

### What Ships
- **Content gap detection** — Ellis queries recipe counts per collection. Collections below a configurable threshold get flagged. This is a DB query + light formatting pass, not a heavy AI call
- **Token budget tracking** — Ellis reads `staff_activity` and sums token usage for the current billing month across all personas
- **Weekly digest endpoint** — `/api/admin/digest` — returns structured JSON: recipes created, submissions reviewed, pending queue count, token spend vs. budget, flagged gaps
- **Weekly Digest card in Owner Dashboard** — Ellis's digest displays on the admin dashboard. Static layout: this week's numbers, gap alerts, budget status
- **Budget config in admin settings** — set monthly token cap. Ellis enforces it starting in Phase 4

### What This Unlocks
- You always know the state of the kitchen at a glance
- Gap detection gives Marco a work queue in Phase 3
- Token visibility means you can tune cost before automation kicks in

### Ellis's Tone in the Dashboard
Operational, no-nonsense. Short bullets. No flourish. Just: here's what happened, here's what needs attention.

## Phase 3 — The Creation Pipeline

*Goal: Marco and Céleste can run batch creation jobs. The full content pipeline — Ellis → creator → Nadia → Theo → Owner Dashboard — works end to end.*

### What Ships
- **Batch creation endpoint** — `/api/admin/staff/create-batch` — accepts a persona (Marco or Céleste), a category/collection target, and a count (3–5). Runs generation, logs to `staff_activity`, returns recipe drafts
- **Nadia's dietary pass** — a dedicated review step that runs after creation. Checks dietary tags for accuracy, flags mislabeled allergens, ensures vegan/GF claims hold up. Structured output: per-tag verdict
- **Theo integration into creation pipeline** — currently Theo (Court of Chefs) only runs on user submissions. Wire him into the creation pipeline too: every Marco/Céleste draft gets a Theo QA pass before landing in the dashboard queue
- **Creation flow trigger in admin** — a "Run Creation Job" button in the admin dashboard. Select persona, select category, confirm count. Manual trigger for now — scheduling comes in Phase 4
- **Pending Queue improvements** — recipe cards in the dashboard now show which staff handled them and at which step, with their notes visible inline

### The Full Flow (End to End)
```
Admin triggers creation job (or Ellis triggers via gap detection)
    ↓
Marco (or Céleste) batch-generates 3–5 recipes
    → Logged to staff_activity
    ↓
Nadia dietary review pass
    → Tags verified / corrected / flagged
    → Logged to staff_activity
    ↓
Theo QA pass
    → Completeness, formatting, safety, step logic
    → Logged to staff_activity
    ↓
Recipe package lands in Owner Dashboard pending queue
    → Shows: creator, Nadia notes, Theo verdict
    ↓
You: Publish ✓ or Reject ✗
```

### Céleste vs. Marco
Céleste is Marco with a different system prompt and a restricted category scope (Baking Alchemy only). Same endpoint, different persona config. One implementation, two crew members.

## Phase 4 — The Kitchen Runs Itself

*Goal: Scheduled jobs. Budget enforcement. Ellis actually triggers Marco instead of you doing it. This is where "autonomous" becomes real.*

### What Ships
- **Cron scheduling** — Vercel Cron (or equivalent) triggers Ellis on a weekly schedule. Ellis checks gaps and dispatches creation jobs to Marco/Céleste automatically. No human action required
- **Budget enforcement**
  - At 80% of monthly cap: scheduled creation runs pause, you get an alert in the dashboard
  - At 95%: all non-essential AI calls pause (scheduled creation only — submissions still process). Banner in admin dashboard. Requires manual override to continue
- **User submission flow fully integrated** — Theo → Nadia → Owner Dashboard. Currently submission review is partially automated. This phase makes it fully consistent with the creation pipeline — same staff, same logging
- **Ellis dispatches to Marco** — creation jobs are now triggered by gap detection, not manual clicks. You can still trigger manually, but you no longer have to

### Scheduling Config (in admin settings)
- Weekly creation run: on/off toggle
- Creation run day/time
- Per-category minimum threshold (below this, Ellis triggers a run)
- Monthly token budget cap

### What "Autonomous" Actually Means
The crew works without you. Recipes are reviewed, created, tagged, and QA'd. They land in your queue. You show up, see a list of vetted work, and approve or reject. That's your entire interaction with the system on a normal week.

## Phase 5 — Full Staff Dashboard

*Goal: The Owner Dashboard becomes the command center. You can see exactly what each crew member did, tune their behavior, and catch quality drift before it affects live content.*

### What Ships
- **Per-persona activity breakdown** — how many actions, token spend, outcomes (pass rate, flag rate, rejection rate) per staff member this month
- **Prompt tuning interface** — each staff member has an editable prompt template in the admin UI. Change Theo's QA criteria, adjust Marco's creation focus, without touching code
- **Rex: system health monitor** — Rex checks: submission pipeline job failures, API error rates, staff member output quality (flag rate trending up = prompt drift signal). Rex's alerts show in the dashboard as a health widget
- **Quality drift alerts** — if Theo's flag rate jumps significantly week over week, Rex surfaces it. If Marco's creations are being rejected more than usual, Rex flags it. You tune the prompt, Rex confirms it's resolved
- **Rejection feedback loop** — when you reject a crew-created recipe with a note, that note is stored and surfaced to the relevant staff member's context on the next run

### Rex's Scope
Rex does not generate content. He reads logs, computes stats, and produces short structured alerts. Minimal token cost. Mostly rule-based with lightweight pattern detection.

## Open Questions Before Phase 1

- [ ] **Token tracking granularity** — do we track per-call or per-run? Per-call is more useful for debugging but more rows. Per-run is simpler to start.
- [ ] **Prompt template storage** — hardcoded in code to start (Phase 1–3), move to DB in Phase 5 when the UI exists to edit them. Confirm this approach.
- [ ] **Cron provider for Phase 4** — Vercel Cron is the natural choice given the stack. Confirm this is available on current Vercel plan.
- [ ] **Nadia's dietary pass** — does she run on every recipe or only when a dietary tag is present? Running on every recipe is safer but costs more tokens. Lean toward: run on every recipe, mark as "no dietary claims" if none found.
- [ ] **Court of Chefs rename** — the user-facing feature is still called "Court of Chefs." Does that brand stay on the profile/submission status page, or does "Reviewed by Theo" replace it publicly? Two valid answers — needs a decision before Phase 3.
